## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 — 신규 회원가입 및 로그인 (우선순위: P1)

신규 사용자가 이메일로 회원가입하거나 카카오 OAuth로 간편 가입한 뒤 로그인하여 서비스를 이용한다.

**이 우선순위인 이유**: 모든 기능의 진입점. 인증 없이는 아무 페이지도 이용할 수 없다.

**독립 테스트**: 회원가입 → 로그인 → 홈 이동만으로 완결된 흐름 검증 가능.

**인수 시나리오**:

1. **Given** 비로그인 사용자가 랜딩 페이지에 있을 때, **When** "시작하기" 버튼을 클릭하면, **Then** 로그인 페이지(`/login`)로 이동한다.
2. **Given** 로그인 사용자가 랜딩 페이지에 있을 때, **When** "시작하기" 버튼을 클릭하면, **Then** 메인 페이지(`/epigrams`)로 이동한다.
3. **Given** 회원가입 폼에서 모든 필드가 유효할 때, **When** "가입하기" 버튼을 클릭하면, **Then** 회원가입이 완료되고 홈(`/`)으로 이동한다.
4. **Given** 이메일 입력창에서 blur 시 비어있으면, **When** 포커스가 이탈하면, **Then** "이메일은 필수 입력입니다." 에러 메시지를 표시한다.
5. **Given** 이메일 형식이 아닐 때 blur 시, **When** 포커스가 이탈하면, **Then** "이메일 형식으로 작성해 주세요." 에러 메시지를 표시한다.
6. **Given** 닉네임이 20자를 초과할 때 blur 시, **When** 포커스가 이탈하면, **Then** "닉네임은 최대 20자까지 가능합니다." 에러 메시지를 표시한다.
7. **Given** 비밀번호가 8자 미만일 때 blur 시, **When** 포커스가 이탈하면, **Then** "비밀번호는 최소 8자 이상입니다." 에러 메시지를 표시한다.
8. **Given** 비밀번호 확인이 비밀번호와 다를 때 blur 시, **When** 포커스가 이탈하면, **Then** "비밀번호가 일치하지 않습니다." 에러 메시지를 표시한다.
9. **Given** 로그인 정보가 올바를 때, **When** 로그인 버튼을 클릭하면, **Then** 홈(`/`)으로 이동한다.
10. **Given** 로그인 정보가 틀렸을 때, **When** 로그인 버튼을 클릭하면, **Then** "이메일 혹은 비밀번호를 확인해주세요." 에러 메시지를 표시한다.
11. **Given** 이미 로그인된 사용자가 `/login`에 접근하면, **When** 페이지가 로드되면, **Then** 홈(`/`)으로 리다이렉트된다.
12. **Given** 카카오 간편 로그인 시 미가입 상태이면, **When** OAuth 콜백이 실행되면, **Then** `/oauth/signup/kakao`로 이동한다.

### 사용자 스토리 2 — 에피그램 메인 페이지 탐색 (우선순위: P1)

로그인한 사용자가 메인 페이지(`/epigrams`)에서 오늘의 에피그램, 최신 에피그램 목록, 최근 댓글을 확인하고 탐색한다.

**이 우선순위인 이유**: 서비스의 핵심 콘텐츠 소비 흐름. 로그인 다음 단계의 주요 경험.

**독립 테스트**: 메인 페이지 진입 → 에피그램 목록 확인 → 더보기 → 상세 이동 흐름으로 독립 검증 가능.

**인수 시나리오**:

1. **Given** 로그인된 사용자가 메인 페이지에 있을 때, **When** 페이지가 로드되면, **Then** 오늘의 에피그램 1개와 최신 에피그램 3개가 표시된다.
2. **Given** 최신 에피그램 섹션에서, **When** "더보기" 버튼을 클릭하면, **Then** 5개의 에피그램이 추가로 표시된다.
3. **Given** 최근 댓글 섹션에서, **When** "더보기" 버튼을 클릭하면, **Then** 4개의 댓글이 추가로 표시된다.
4. **Given** 오늘의 감정을 아직 선택하지 않은 사용자가, **When** 감정(감동/기쁨/고민/슬픔/분노) 중 하나를 선택하면, **Then** 감정 선택 컴포넌트가 사라진다.
5. **Given** 이미 오늘의 감정을 선택한 사용자가, **When** 메인 페이지에 접근하면, **Then** 감정 선택 컴포넌트가 렌더링되지 않는다 (완전히 숨김).
6. **Given** 사용자가 일정 수준 스크롤을 내리면, **When** 위화살표 버튼이 나타나고 클릭하면, **Then** 페이지 최상단으로 이동한다.
7. **Given** 에피그램 카드를 클릭하면, **When** 클릭 시, **Then** 해당 에피그램 상세 페이지(`/epigrams/:id`)로 이동한다.

### 사용자 스토리 3 — 에피그램 작성 (우선순위: P1)

로그인한 사용자가 에피그램 작성 페이지(`/addepigram`)에서 새 에피그램을 작성하고 저장한다.

**이 우선순위인 이유**: 서비스의 핵심 CRUD 중 Create. 콘텐츠 생산의 시작점.

**독립 테스트**: 작성 폼 입력 → 저장 → 상세 페이지 이동으로 독립 검증 가능.

**인수 시나리오**:

1. **Given** 내용이 500자를 초과하면, **When** 저장 버튼을 누르면, **Then** 저장이 불가하고 입력창 하단에 경고 메시지가 표시된다.
2. **Given** 저자 항목에서 '직접입력'을 선택하면, **When** 선택하면, **Then** 직접 텍스트 입력창이 활성화된다.
3. **Given** 태그가 이미 3개 등록된 상태에서, **When** 태그를 추가하려 하면, **Then** 추가가 불가하다.
4. **Given** 한 태그가 10자를 초과하면, **When** 태그를 입력하면, **Then** 10자까지만 입력 가능하다.
5. **Given** 모든 필수 항목이 유효하게 입력된 경우, **When** 저장 버튼을 클릭하면, **Then** 에피그램이 저장되고 해당 상세 페이지(`/epigrams/:id`)로 이동한다.

### 사용자 스토리 4 — 에피그램 상세 조회 및 반응 (우선순위: P2)

사용자가 에피그램 상세 페이지(`/epigrams/:id`)에서 에피그램 내용을 보고, 좋아요·댓글을 남기거나 수정·삭제한다.

**이 우선순위인 이유**: 상세 조회와 반응 기능은 커뮤니티 플랫폼의 핵심 상호작용이지만, 작성·탐색 이후에 완전한 의미를 갖는다.

**독립 테스트**: 상세 페이지 직접 URL 진입 → 좋아요 토글 → 댓글 작성으로 독립 검증 가능.

**인수 시나리오**:

1. **Given** 에피그램 상세 페이지에서, **When** "공유" 버튼을 클릭하면, **Then** 현재 URL이 클립보드에 복사된다.
2. **Given** 출처에 URL이 있는 에피그램에서, **When** 새창 버튼을 클릭하면, **Then** 해당 URL이 새창으로 열린다.
3. **Given** 자신이 작성한 에피그램에서, **When** "…" 버튼을 클릭하면, **Then** 수정/삭제 메뉴가 나타난다.
4. **Given** 삭제를 선택하면, **When** 확인 모달에서 삭제를 누르면, **Then** 에피그램이 삭제되고 메인 페이지(`/epigrams`)로 이동한다.
5. **Given** 댓글 입력창에 내용을 작성하고, **When** 저장 버튼을 클릭하면, **Then** 댓글이 등록되고 목록에 추가된다.
6. **Given** 댓글 공개/비공개 토글에서, **When** 비공개를 선택하면, **Then** 해당 댓글은 작성자 본인에게만 보인다 (기본값: 공개).
7. **Given** 댓글 목록에서 스크롤을 내리면, **When** 하단에 도달하면, **Then** 추가 댓글이 무한 스크롤로 로드된다.
8. **Given** 자신이 작성한 댓글에서 수정을 선택하면, **When** 수정 버튼을 클릭하면, **Then** 해당 댓글 자리에서 인라인으로 textarea가 활성화되어 내용을 수정할 수 있다.
9. **Given** 태그를 클릭하면, **When** 클릭 시, **Then** 해당 키워드로 검색 페이지(`/search?keyword=태그명`)로 이동한다.
10. **Given** 댓글 작성자의 프로필 이미지 또는 아이디를 클릭하면, **When** 클릭 시, **Then** 해당 사용자 프로필이 모달로 표시된다.

### 사용자 스토리 5 — 에피그램 검색 (우선순위: P2)

사용자가 검색 페이지(`/search`)에서 에피그램을 검색하고 결과를 탐색한다.

**이 우선순위인 이유**: 콘텐츠 탐색의 핵심 경로이지만, 콘텐츠 생산 이후에 완전한 가치를 갖는다.

**독립 테스트**: 검색어 입력 → 결과 표시 → 결과 클릭 → 상세 이동으로 독립 검증 가능.

**인수 시나리오**:

1. **Given** 검색어를 입력하고 검색하면, **When** 결과가 표시될 때, **Then** 검색 키워드가 결과 텍스트 내에서 하이라이팅된다.
2. **Given** 검색 결과가 있을 때 스크롤을 내리면, **When** 하단에 도달하면, **Then** 추가 결과가 무한 스크롤로 로드된다.
3. **Given** 검색어가 있는 상태에서 새로고침하면, **When** 페이지가 로드되면, **Then** URL의 쿼리 파라미터에서 검색어를 복원하여 결과를 다시 표시한다.
4. **Given** 최근 검색어가 있을 때, **When** "모두 지우기"를 클릭하면, **Then** 최근 검색어 섹션이 제거된다.
5. **Given** 최근 검색어 항목을 클릭하면, **When** 클릭 시, **Then** 해당 검색어로 다시 검색한다.
6. **Given** 검색 결과 아이템을 클릭하면, **When** 클릭 시, **Then** 해당 에피그램 상세 페이지로 이동한다.

### 사용자 스토리 6 — 마이페이지 (우선순위: P3)

사용자가 마이페이지(`/mypage`)에서 자신의 활동 현황을 확인하고, 프로필을 관리하며, 자신의 에피그램과 댓글을 탐색한다.

**이 우선순위인 이유**: 데이터 시각화(달력, 차트)와 프로필 관리는 핵심 기능이 완성된 이후에 부가 가치를 제공한다.

**독립 테스트**: 마이페이지 직접 진입 → 감정 달력 확인 → 내 에피그램 클릭 → 상세 이동으로 독립 검증 가능.

**인수 시나리오**:

1. **Given** 마이페이지에서, **When** "로그아웃" 버튼을 클릭하면, **Then** 로그아웃되고 홈(`/`)으로 이동한다.
2. **Given** 프로필 이미지를 클릭하면, **When** 파일 선택 창에서 영문 파일명 이미지를 선택하면, **Then** 이미지가 업로드되고 프로필에 반영된다.
3. **Given** 감정 달력에서, **When** 월을 이동하면, **Then** 해당 월의 날짜별 감정 이모지가 달력에 표시된다.
4. **Given** 감정 파이 차트에서, **When** 해당 월의 감정 데이터가 있으면, **Then** 감동/기쁨/고민/슬픔/분노 비율이 파이 차트로 표시된다.
5. **Given** 내 에피그램 목록에서 항목을 클릭하면, **When** 클릭 시, **Then** 해당 에피그램 상세 페이지로 이동한다.
6. **Given** 내 댓글 목록 제목에서, **When** 페이지가 로드되면, **Then** 총 댓글 수가 괄호 안에 표시된다.
7. **Given** 내 댓글 항목을 클릭하면, **When** 클릭 시, **Then** 해당 댓글이 있는 에피그램 상세 페이지로 이동한다.

### 엣지 케이스

- 닉네임 중복 가입 시 서버에서 500 에러가 반환되므로, 이를 "이미 사용 중인 닉네임입니다."로 사용자에게 표시한다.
- 이미지 파일명에 한글이 포함된 경우 업로드 오류가 발생하므로, 파일 선택 시 또는 업로드 실패 시 영문 파일명 사용을 안내한다.
- 로그인 상태에서 `/login`, `/signup`, `/oauth/signup/kakao` 접근 시 홈(`/`)으로 즉시 리다이렉트한다.
- 댓글/에피그램 무한 스크롤에서 `nextCursor`가 `null`이면 더 이상 로드하지 않고 트리거를 숨긴다.
- 오늘의 감정은 하루에 한 번만 선택 가능하다. `GET /{teamId}/emotionLogs/today`로 오늘 감정 존재 여부를 확인한다.
- OAuth 간편 가입 시 회원가입이 이미 되어 있으면 바로 로그인 처리, 미가입 시 `/oauth/signup/kakao`로 이동한다.
- 에피그램이 없는 사용자의 마이페이지에서 "내 에피그램" 섹션은 빈 상태 UI를 표시한다.

## 요구사항 *(필수)*

### 공통 기능 요구사항

- **FR-001**: 시스템은 반드시 피그마 시안의 컬러 팔레트와 폰트 시스템을 전역 설정으로 정의해야 한다.
- **FR-002**: 시스템은 재사용 가능한 공용 컴포넌트(버튼, 입력창, 모달, 태그 등)를 구현해야 한다.
- **FR-003**: 시스템은 반드시 반응형 웹 디자인을 지원해야 한다 (모바일 360px · 태블릿 744px · 데스크톱 1920px 기준).
- **FR-004**: 상단 네비게이션바는 로고(→ `/`), 검색 아이콘(→ `/search`), 사람 아이콘(→ `/mypage`) 링크를 포함해야 한다.

### BFF 프록시 요구사항

백엔드 API는 로그인/회원가입 응답 body에 JWT 토큰을 반환한다. 클라이언트에서 이를 직접 다루면 XSS 공격에 취약하므로, Next.js API Route를 BFF(Backend For Frontend) 프록시로 구성하여 보안을 강화한다.

- **FR-008**: 클라이언트의 모든 API 요청은 외부 백엔드에 직접 닿지 않고 반드시 BFF 프록시(`/api/[...path]` catch-all 라우트)를 경유해야 한다.
- **FR-009**: BFF 프록시는 다음 역할을 수행해야 한다:
  - 로그인·회원가입 요청 시 백엔드가 반환한 응답 body의 `accessToken`, `refreshToken`을 꺼내어 `HttpOnly; Secure; SameSite=Strict` 쿠키로 클라이언트에 내려준다. 토큰 값은 클라이언트 JavaScript에 노출되지 않아야 한다.
  - 이후 모든 인증이 필요한 요청에서 쿠키의 `accessToken`을 꺼내 백엔드에 `Authorization: Bearer` 헤더로 자동 첨부한다.
  - `accessToken` 만료(401 응답) 시 프록시 내부에서 `POST /{teamId}/auth/refresh-token`을 호출하여 토큰을 갱신하고, 원래 요청을 재시도한 뒤 새 토큰을 쿠키로 내려준다. 클라이언트는 토큰 갱신 여부를 인식하지 않아도 된다.
  - `refreshToken`도 만료된 경우 쿠키를 삭제하고 401을 클라이언트에 반환하여 로그아웃 처리를 유도한다.
  - 로그아웃 시 `accessToken`, `refreshToken` 쿠키를 모두 삭제한다.

### 인증 요구사항

- **FR-010**: 일반 회원가입은 이메일, 닉네임, 비밀번호, 비밀번호 확인 필드를 포함해야 한다. (BFF 프록시를 통해 `POST /{teamId}/auth/signUp` 호출)
- **FR-011**: 폼 검증은 blur 이벤트 시점에 실행되어야 한다. 유효성 검사 규칙은 아래와 같다:
  - 이메일: 필수, 이메일 형식
  - 닉네임(일반 가입): 필수, 최대 20자
  - 닉네임(OAuth 가입): 필수, 최대 10자
  - 비밀번호: 필수, 최소 8자, 숫자·영문·특수문자로만 구성
  - 비밀번호 확인: 필수, 비밀번호와 일치
- **FR-012**: 로그인 실패 시 이메일 입력창에 "이메일 혹은 비밀번호를 확인해주세요." 메시지를 표시하고, 두 입력창을 에러 상태로 표시해야 한다. (BFF 프록시를 통해 `POST /{teamId}/auth/signIn` 호출)
- **FR-013**: 카카오 OAuth 로그인을 지원해야 한다. (BFF 프록시를 통해 `POST /{teamId}/auth/signIn/kakao` 호출)
- **FR-014**: 인증 토큰(accessToken, refreshToken)은 BFF 프록시가 설정하는 HttpOnly 쿠키에 저장되어야 하며, 클라이언트 코드에서 직접 토큰에 접근하거나 localStorage에 저장하는 것을 금지한다.
- **FR-015**: accessToken 갱신 로직은 BFF 프록시 내부에서 처리되어야 하며, 클라이언트는 토큰 갱신을 인식하지 않아도 정상 동작해야 한다.
- **FR-016**: 인증이 필요한 페이지는 보호되어야 하며, 비로그인 접근 시 `/login`으로 리다이렉트한다.
- **FR-017**: 로그인 상태에서 인증 페이지(`/login`, `/signup`, `/oauth/signup/kakao`) 접근 시 홈(`/`)으로 리다이렉트한다.

### 에피그램 메인 페이지 요구사항

- **FR-020**: 오늘의 에피그램은 `GET /{teamId}/epigrams/today`로 가져와 표시해야 한다.
- **FR-021**: 오늘의 감정 데이터 유무를 `GET /{teamId}/emotionLogs/today`로 확인하고, 미등록 시 감정 선택 UI를 표시해야 한다. 이미 등록된 경우 감정 선택 컴포넌트를 렌더링하지 않는다 (완전 숨김).
- **FR-022**: 감정 선택(감동/기쁨/고민/슬픔/분노) 시 `POST /{teamId}/emotionLogs/today`로 저장하고, 선택 컴포넌트를 즉시 숨겨야 한다.
- **FR-023**: 최신 에피그램은 초기 3개를 표시하며(`GET /{teamId}/epigrams?limit=3`), 더보기 클릭마다 5개씩 추가 로드한다 (cursor 기반).
- **FR-024**: 최근 댓글은 초기 표시 후 더보기 클릭마다 4개씩 추가 로드한다 (`GET /{teamId}/comments?limit=4`, cursor 기반).
- **FR-025**: 스크롤 위치가 일정 이상 내려가면 위화살표 버튼을 표시하며, 클릭 시 페이지 최상단으로 이동해야 한다.

### 에피그램 작성/수정 요구사항

- **FR-030**: 에피그램 내용은 최대 500자로 제한하며, 초과 시 저장 버튼을 비활성화하고 경고 메시지를 표시해야 한다.
- **FR-031**: 저자 항목은 '직접입력', '알 수 없음', '본인' 라디오 버튼을 제공해야 하며 필수 입력이다. `author` 필드로 API에 전달한다.
- **FR-032**: 출처는 제목(`referenceTitle`)과 URL(`referenceUrl`) 필드로 구성되며, 선택 입력이다.
- **FR-033**: 태그는 최대 3개, 태그당 최대 10자로 제한해야 한다. `tags` 배열로 API에 전달한다.
- **FR-034**: 에피그램 작성은 `POST /{teamId}/epigrams`, 수정은 `PATCH /{teamId}/epigrams/{id}`를 사용한다. 수정 페이지(`/epigrams/:id/edit`) 진입 시 기존 데이터를 폼에 미리 채워야 한다.
- **FR-035**: 저장 완료 후 해당 에피그램 상세 페이지(`/epigrams/:id`)로 이동해야 한다.
- **FR-036**: 수정 페이지에서 취소 버튼 클릭 시 해당 에피그램 상세 페이지(`/epigrams/:id`)로 이동해야 한다.

### 에피그램 상세 페이지 요구사항

- **FR-040**: 에피그램 데이터는 `GET /{teamId}/epigrams/{id}`로 가져오며, 내용·저자·출처·태그·좋아요 수를 표시해야 한다.
- **FR-041**: 공유 버튼 클릭 시 현재 URL을 클립보드에 복사해야 한다.
- **FR-042**: 좋아요 추가는 `POST /{teamId}/epigrams/{id}/like`, 취소는 `DELETE /{teamId}/epigrams/{id}/like`를 사용한다.
- **FR-043**: 자신이 작성한 에피그램(`writerId === 현재 사용자 id`)에만 수정/삭제 메뉴를 표시해야 한다.
- **FR-044**: 에피그램 삭제는 `DELETE /{teamId}/epigrams/{id}`이며, 확인 모달을 통해 실행해야 한다. 삭제 완료 후 메인 페이지(`/epigrams`)로 이동한다.
- **FR-045**: 댓글 목록은 `GET /{teamId}/epigrams/{id}/comments?limit=N&cursor=N`으로 무한 스크롤 구현해야 한다.
- **FR-046**: 댓글 작성은 `POST /{teamId}/comments`이며, 공개/비공개 토글(기본값: 공개, `isPrivate: false`)을 제공해야 한다.
- **FR-047**: 자신이 작성한 댓글에 대해 수정(`PATCH /{teamId}/comments/{id}`)·삭제(`DELETE /{teamId}/comments/{id}`) 기능을 제공해야 한다. 수정은 인라인 편집 방식으로, 해당 댓글 자리에서 textarea가 활성화되며 저장/취소 버튼 두 개를 함께 표시한다. 취소 버튼 클릭 시 수정 내용을 버리고 원래 댓글 내용으로 복원한다.
- **FR-048**: 댓글 작성자의 프로필 이미지 또는 아이디 클릭 시 `GET /{teamId}/users/{id}`로 사용자 정보를 가져와 모달로 표시해야 한다.

### 검색 페이지 요구사항

- **FR-050**: 에피그램 검색은 `GET /{teamId}/epigrams?keyword={검색어}&limit=N&cursor=N`을 사용하며, 무한 스크롤을 구현해야 한다.
- **FR-051**: 검색어는 URL 쿼리 파라미터(`?keyword=...`)로 유지하여 새로고침 시에도 검색 결과를 복원해야 한다.
- **FR-052**: 검색 결과에서 검색 키워드를 텍스트 내에서 하이라이팅하여 표시해야 한다.
- **FR-053**: 최근 검색어는 localStorage에 저장해야 하며, 클릭 시 재검색, "모두 지우기" 클릭 시 전체 삭제해야 한다.

### 마이페이지 요구사항

- **FR-060**: 내 프로필 정보는 `GET /{teamId}/users/me`로 가져와 닉네임·프로필 이미지를 표시해야 한다.
- **FR-061**: 프로필 이미지 변경은 `POST /{teamId}/images/upload`로 업로드한 뒤, `PATCH /{teamId}/users/me`로 이미지 URL을 저장해야 한다. 파일명은 영문이어야 하며, 한글 파일명에 대한 안내를 제공해야 한다.
- **FR-062**: 감정 달력은 `GET /{teamId}/emotionLogs/monthly?userId=N&year=N&month=N`으로 월별 데이터를 가져와 달력 UI로 표시해야 한다.
- **FR-063**: 감정 파이 차트는 월별 감정 데이터를 집계하여 감동/기쁨/고민/슬픔/분노 비율을 파이 차트로 표시해야 한다.
- **FR-064**: 내 에피그램 목록은 `GET /{teamId}/epigrams?writerId={내 id}&limit=N&cursor=N`으로 가져오며, 더보기 버튼 클릭 시 추가 로드하는 방식으로 페이지네이션을 구현한다.
- **FR-065**: 내 댓글 목록은 `GET /{teamId}/users/{id}/comments?limit=N&cursor=N`으로 가져오며, 더보기 버튼 클릭 시 추가 로드하는 방식으로 페이지네이션을 구현한다. 총 댓글 수를 제목 옆 괄호에 표시해야 한다.
- **FR-066**: 로그아웃 시 인증 토큰을 삭제하고 홈(`/`)으로 이동해야 한다.

## 가정 사항

- 사용자는 인터넷에 연결된 환경에서 서비스를 이용한다.
- 백엔드 API의 `{teamId}` 경로 파라미터는 환경 변수(`NEXT_PUBLIC_TEAM_ID`)로 관리하며, 모든 요청에 자동으로 포함된다.
- 카카오 OAuth를 우선 구현하며, 구글·네이버는 UI는 존재하지만 실제 연동은 추후 확장한다.
- 데이터 시각화(달력, 파이 차트) 외부 라이브러리는 구현 시점에 팀에서 선정한다 (후보: `react-calendar`, `recharts`, `chart.js` 등).
- 이미지 업로드 시 파일명은 반드시 영문이어야 하며, 한글 파일명 업로드 시 API 오류가 발생한다.
- 피그마 시안의 컬러·타이포그래피·컴포넌트 스타일을 우선 기준으로 삼는다. Figma 파일(`tk2V6CQ7SYVsFfdIPF1EW8`)에 대한 뷰 접근 권한이 필요하다.
- 댓글·에피그램 페이지네이션은 cursor 기반이며, 응답의 `nextCursor`가 `null`이면 마지막 페이지로 간주한다.
- React Query의 `useInfiniteQuery`를 사용하여 cursor 기반 무한 스크롤을 구현한다.
